# Phase B — Demo Data & Account Strategy

## Status

PLANNED / READY FOR EXECUTION

Phase B is not complete yet because this phase has not created real staging data. This document is the Phase B1 plan for demo data and account strategy.

## Purpose

Phase 8 and Phase 9 validated the staging deployment and formalized the staging release roadmap. The current staging content still comes from smoke/QA work, so it proves the app works but is not polished enough for a portfolio or long-term demo.

Phase B turns the project direction from "staging works" into "staging has a safe, intentional demo-quality content strategy."

## Current Problem

- Staging contains functional smoke data.
- Smoke data is useful for QA evidence.
- Smoke data is not suitable for portfolio/demo presentation.
- Names such as "Phase 8.8 Smoke Question" should not remain as the visible long-term demo content.
- Demo content must look intentional, educational, and clean.

## Demo Account Strategy

### Option 1 — No Public Demo Account

Description:
Visitors can browse public quizzes as guests and may register their own account if they want to try the user flow.

Pros:

- Safest.
- No shared credentials.
- No risk of public password abuse.
- No demo account cleanup problem.

Cons:

- Recruiters/testers must register before trying user flow.
- Slightly more friction.

### Option 2 — Public Demo User Account

Description:
A limited USER account is created for demo usage.

Pros:

- Easier portfolio demo.
- Testers can try result/review/history flow quickly.

Cons:

- Shared password can be abused.
- Attempt history becomes messy.
- Password should not be committed to repository.
- Requires periodic cleanup or reset.

### Option 3 — Public Demo Admin Account

Description:
A public ADMIN account is created.

Recommendation:
Do not use this for public portfolio.

Reason:
Admin can create/edit/delete content, publish/unpublish quizzes, and damage demo data. Public admin credentials are too risky even on staging.

### Recommended Decision

- No public demo admin account.
- Keep admin account private owner-only.
- Allow public guest browsing.
- Allow public self-registration if current app behavior supports it.
- Optional private demo user account may exist, but its password must not be stored in repository docs.
- Do not publish admin credentials in README, GitHub, screenshots, docs, or chat logs.
- Do not store demo passwords in committed files.

If a demo user is created later, document only the existence and purpose of the account, not the password.

## Data Creation Options

### Option A — Manual Creation Through Admin UI

Recommended for initial v1.0 demo data.

Pros:

- Safest.
- Exercises real admin workflow.
- Avoids SQL/schema mistakes.
- Avoids accidental production seed behavior.
- Good for verifying actual UI usability.

Cons:

- Slower.
- Manual work.
- Needs private admin access.

### Option B — Staging-Only SQL Script

Use only if manual UI creation becomes too slow or unreliable.

Pros:

- Repeatable.
- Faster reset.

Cons:

- Risky if schema assumptions are wrong.
- Can bypass application validation.
- Must never run against production.
- Must not contain secrets.

### Option C — Staging-Only Seed Script

Use later only if there is a clear need for repeatable demo reset.

Pros:

- Repeatable.
- Can be integrated into controlled staging setup.

Cons:

- Easy to accidentally enable in the wrong environment.
- Requires careful environment guards.
- More code and testing.

### Recommended Decision

For v1.0:

- Use Admin UI manual creation first.
- Do not create or enable seed script yet.
- Do not run SQL unless explicitly approved later.
- Document the dataset and creation process.
- Preserve smoke data unless there is a safe reason to remove it.
- Avoid destructive cleanup until demo data is verified.
